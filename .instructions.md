# Project Instructions — TCC Simple Pipeline

## Terminal & Execution Rules

### ⛔ NEVER use PowerShell directly
- All terminal commands must go through **VS Code tasks** (`.vscode/tasks.json`).
- Use `run_vscode_command` with `workbench.action.terminal.runSelectedText` or create/run tasks.
- If a task doesn't exist yet, **create it in `tasks.json` first**, then run it.

### ⛔ NEVER type secrets or passwords in prompts
- If a command requires a password/passphrase, tell the user to type it directly in the terminal.
- Never use `vscode_askQuestions` for secrets.

### ✅ Python execution
- Make sure we are in a uv venv. Always keep it active
- Use uv add and uv sync (uv is the python package manager and virtual environment creator that substitutes pip and venv modules) tasks to install python dependencies before running any Python for the first time.
- Prefer running Python via VS Code tasks that invoke the project's venv. or through Pylance MCP Python snippets
- Use Pylance MCP tools for type checking and diagnostics.

### ✅ Git & GitHub
- Git operations: use the GitHub MCP tools and VS Code tasks as an alternative.
- Do **not** run `git` commands directly in PowerShell.
- Create tasks in `.vscode/tasks.json` for common git operations (branch, commit, push).
- For PRs, issues, and repo operations, use the GitHub MCP tools.

## Code Conventions

### Python
- **Python 3.11+**, type hints required (`from __future__ import annotations`).
- **Polars** for DataFrame operations (never Pandas).
- **SQLAlchemy** for Oracle connections via `oracledb` thin mode.
- CSVs use `;` as delimiter and `utf-8-sig` encoding.
- All file paths use `pathlib.Path`, never `os.path`.
- Dataclasses for structured return types (`@dataclass(frozen=True)` when possible).
- `_log()` helper for pipeline progress messages.

### File Organization
- `pipeline.py` — daily production pipeline (Polars, Parquet)
- `araucaria_sample_pipeline.py` — standalone sample pipeline (200 NIOs, CSV)
- `db.py` — database engine factory
- `queries/` — SQL files, parameterized with `:BINDS`
- `output/` — all generated data (gitignored)
- `tests/` — pytest test files
- `docs/` — planning and documentation
- `scripts/` — standalone utility scripts

### Testing
- Tests live in `tests/` directory.
- Use `pytest` as the test runner.
- Test files: `test_<module_name>.py`.
- Use `tmp_path` fixture for filesystem tests (no real DB needed).
- Mock database connections where possible.

### Naming
- Files: `snake_case.py`
- Functions: `snake_case()`
- Classes: `PascalCase()`
- Constants: `UPPER_SNAKE_CASE`
- Private helpers: `_leading_underscore()`
